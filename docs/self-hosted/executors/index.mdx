# Deploy executors

Executors run resource-intensive or untrusted jobs for precise code navigation and Batch Changes. This guide helps self-hosted operators choose a deployment method and prepare the infrastructure that executors require.

For feature concepts, Sourcegraph configuration, connected instance management, and executor secrets, see [Administer executors](/admin/executors).

## Before you deploy

All executor deployments require:

- A Sourcegraph Enterprise instance running the same major and minor version as the executor
- Network connectivity from the executor to the Sourcegraph frontend
- An executor authentication secret configured in Sourcegraph and on the executor
- Linux AMD64 compute for the executor runtime
- Sufficient CPU, memory, and disk for the number of concurrent jobs

### Configure authentication

Before starting an executor, generate a random secret containing at least 20 characters. Set the secret as `executors.accessToken` under **Site admin > Configuration > Advanced configuration**.

Configure the same value on the executor as `EXECUTOR_FRONTEND_PASSWORD`, or through the equivalent input for its deployment method. See [Configure executor authentication](/admin/executors#configure-executor-authentication) for the complete administrator workflow.

### Size executor resources

Plan for the following resources for each concurrent job:

| Resource | Per concurrent job |
| -------- | -----------------: |
| CPU      |            4 cores |
| Memory   |              12 GB |
| Disk     |              20 GB |

For example, an executor configured to process four jobs concurrently should have approximately 16 CPU cores, 48 GB of memory, and 80 GB of disk. Configure concurrency with `EXECUTOR_MAXIMUM_NUM_JOBS`, and adjust per-job CPU and memory with `EXECUTOR_JOB_NUM_CPUS` and `EXECUTOR_JOB_MEMORY`.

On AWS, use 125 MiB/s disk throughput and provision IOPS appropriate for the disk size. The smallest AWS instance that supports Firecracker is currently `c5n.metal`, so increase concurrency when appropriate to make use of the available capacity.

See the [executor configuration reference](/self-hosted/executors/executors-config) for all runtime settings.

## Choose a deployment method

The primary decision is whether jobs require a virtual-machine isolation boundary.

- **Use Firecracker for the strongest supported job isolation.** Choose a Terraform deployment on AWS or Google Cloud, or install the Linux binary on compatible infrastructure. Firecracker requires KVM: AWS uses bare-metal instances, while Google Cloud supports nested virtualization.
- **Use Docker when KVM is unavailable and privileged container execution is acceptable.** Install the Linux binary in Docker mode or use Docker Compose.
- **Use Kubernetes only when executor workloads must run in an existing cluster.** Docker-in-Docker and native Kubernetes executors are beta deployment methods with weaker isolation and known production reliability limitations. Native Kubernetes executors are being deprecated.

<Callout type="warning">
	Executors run untrusted repository code and build tooling. Deployment
	methods without Firecracker do not provide a MicroVM boundary. Isolate
	executor hosts or nodes from sensitive workloads, restrict access to private
	networks and cloud metadata services, and use disposable compute where
	possible.
</Callout>

## Configure and validate your deployment

After choosing a deployment method, follow its guide for installation and method-specific validation. Then confirm that the executor is active and reporting the expected version under **Site admin > Maintenance > Executor instances**.
